# الفرق بين Skill وFramework

هذا الملف يشرح بلغة بسيطة جدًا لماذا نسمي هذا المشروع **"Arabic Legal Research Skill Framework"**
وليس مجرد "Skill" أو "Prompt".

## أولًا: ما معنى "Skill"؟

**Skill** هي ملف تعليمات (مثل `SKILL.md` في هذا المستودع) تُعطى لنموذج ذكاء اصطناعي مثل Claude أو
ChatGPT، فيقرأها ويحاول الالتزام بها في كل محادثة. الـ Skill بمفردها:

- تعتمد فقط على النموذج اللغوي لتفسيرها وتنفيذها بأمانة.
- لا يوجد كود ينفّذ فحصًا حقيقيًا على الناتج.
- لا تملك وسيلة لإثبات أن الناتج التزم فعلًا بالقواعد — فقط "تعليمات ويُفترض الالتزام".
- عرضة لتفاوت السلوك بين محادثة وأخرى، أو بين نموذج وآخر.

## ثانيًا: ما الذي يضيفه "Framework"؟

**Framework** في هذا المشروع يعني: تعليمات الـ Skill + **طبقة تنفيذية حقيقية** تتحقق من النتائج
آليًا، بدل الاكتفاء بالثقة في التزام النموذج. تحديدًا:

| الطبقة | ماذا تضيف | أين تجدها |
|---|---|---|
| 📜 **تعليمات الـ Skill** | القواعد المنهجية والأسلوبية التي يقرأها النموذج | `SKILL.md`, `rules/`, `checklists/`, `validators/` |
| 🧬 **مخطط بيانات صارم (Schema)** | يرفض أي بيانات بحث لا تطابق البنية المطلوبة | `schemas/*.json` (JSON Schema Draft 2020-12) |
| ⚙️ **محرك تحقق تنفيذي (CLI)** | يفحص فعليًا: التسلسل الهرمي، الاستشهادات، الحواشي، حماية الخطة المعتمدة، تعارض الأولويات | `src/legal_research_skill/validators/` عبر أمر `legal-research-skill validate` |
| 📄 **مولّد ومدقق DOCX حقيقي** | ينتج ملف Word فعليًا ويفحص بنيته (OPC/OOXML) بدل افتراض أنه صحيح | `src/legal_research_skill/docx/` عبر `render-docx` و`validate-docx` |
| 🪟 **بوابة Word اختيارية** | تفتح الملف داخل Word نفسه، تُحدّث الحقول، تحفظ، وتعيد الفحص — فقط عندها يُقال `WORD_VALIDATED` | `src/legal_research_skill/word/` عبر `finalize-word --require-word` |
| 🧪 **اختبارات آلية** | 233+ اختبار يمنع أي تراجع صامت في السلوك | `tests/` |
| 🏗️ **بناء وتوزيع قابل لإعادة الإنتاج** | حزمة Python حقيقية، CI على GitHub Actions، بناء نسخة (wheel) بنفس المحتوى دائمًا | `pyproject.toml`, `.github/workflows/` |

## ثالثًا: لماذا هذا الفرق مهم عمليًا؟

لو استخدمت "Skill" وحدها، فأنت تثق أن النموذج **لن ينسى قاعدة ولن يخترع مصدرًا** — وهذا افتراض
غير مضمون بطبيعة النماذج اللغوية. أما مع الـ **Framework**، يمكنك أن تُشغّل فحصًا تنفيذيًا مستقلًا
عن النموذج نفسه:

```bash
legal-research-skill validate my-research.json --fail-on high
legal-research-skill validate-docx draft.docx --format json
```

هذه الأوامر لا "تسأل" نموذجًا هل التزم — هي **كود يفحص الملف فعليًا** ويعطيك نتيجة حتمية
(pass/fail) قابلة لإعادة الإنتاج بنفس النتيجة كل مرة.

## رابعًا: هل يمكن استخدام الجزء الأول فقط (Skill بدون تثبيت)؟

نعم تمامًا — إن كنت مستخدمًا عاديًا لا يريد تثبيت Python، يمكنك استخدام `SKILL.md` مباشرة داخل
ChatGPT أو Claude بدون أي تثبيت (راجع [`docs/quickstart.md`](quickstart.md)). لكن إن كنت تريد
تحققًا تنفيذيًا حقيقيًا من مخرجاتك، أو تبني نظامًا آليًا فوق هذا المشروع، فأنت تحتاج طبقة
Framework الكاملة عبر تثبيت الحزمة (راجع قسم "التثبيت للمطورين" في [`README.md`](../README.md)).

## خامسًا: ما الذي لا يزال يعتمد على النموذج اللغوي؟

الـ Framework **لا يلغي** الحاجة إلى نموذج ذكاء اصطناعي جيد الالتزام — هو يقلل الاعتماد الأعمى
عليه فقط في النقاط القابلة للفحص الآلي. الصياغة، الفهم القانوني للموضوع، وجودة التحليل تبقى من
مسؤولية النموذج والمستخدم معًا. راجع [`docs/limitations.md`](limitations.md) لحدود دقيقة لما
يفحصه الكود فعلًا وما لا يفحصه.
