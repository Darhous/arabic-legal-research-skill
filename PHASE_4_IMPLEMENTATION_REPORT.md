## 2. Previous Hang

The prior implementation called Microsoft Word COM automation in-process from the CLI. A real smoke command using:

```powershell
python -m legal_research_skill finalize-word tests_tmp/sample.docx --output tests_tmp/sample.word.docx --format json
```

hung for several hours before manual interruption.

Root cause: Word COM automation can block indefinitely when invoked directly from the CLI process. The path had no timeout isolation and the smoke command used an older `tests_tmp/sample.docx` instead of the DOCX generated by the latest `build-artifact`.

Correction: Word finalization now runs through a subprocess worker invoked by `src/legal_research_skill/word/runner.py`, with a configurable timeout. The CLI process no longer calls `DispatchEx` directly.

## 4. Worker Subprocess Design

Word automation is split into:

- `worker.py`: performs Windows/pywin32/Word COM work with `DispatchEx("Word.Application")`, invisible Word, disabled alerts, field update, TOC update, repagination, save, close, reopen, and quit.
- `worker_cli.py`: minimal JSON stdin-free worker entrypoint called as `python -m legal_research_skill.word.worker_cli <input> <working>`.
- `runner.py`: starts the worker with `subprocess.Popen`, reads JSON output, and enforces timeout with `communicate(timeout=...)`.
- `finalization.py`: validates paths, copies input to a temp DOCX, calls the runner, validates the post-Word DOCX structurally, and atomically replaces final output only after success.

The worker uses `finally` blocks to call `document.Close(False)` and `app.Quit()` when it controls those objects.

## 5. Timeout Behavior

Default timeout:

- `60` seconds.

Bounds:

- Minimum: `5` seconds.
- Maximum: `300` seconds.

CLI option:

```powershell
--word-timeout-seconds 60
```

On timeout, only the worker subprocess is terminated. If it does not exit after a short wait, it is killed. The code does not call `taskkill`, does not match by `WINWORD.EXE` name, and does not terminate general user Word sessions.

Timeout result shape includes:

- `status: "TIMEOUT"`
- `error_code: "word_timeout"`
- `timeout_seconds`
- `word_validated: false`

## 6. Word Process Ownership

The implementation avoids killing global Word sessions. It records the worker PID in timeout evidence and terminates only that worker process.

During cleanup inspection, a `WINWORD.EXE` process and an older `python.exe` process existed with start times around the interrupted run. CIM access was denied and `wmic` was unavailable, so ownership could not be proven. They were not terminated automatically.

Word statuses include:

- `WORD_VALIDATED`
- `NOT_AVAILABLE`
- `NOT_RUN`
- `TIMEOUT`
- `FAILED`
- `BLOCKED` at artifact level when Word is required and cannot complete

CLI exit codes:

- `0`: requested operation succeeded.
- `1`: validation or artifact failure.
- `2`: input, argument, configuration, or unsafe path error.
- `3`: required external Word gate unavailable, timed out, not run, or blocked.
- `4`: unexpected Word automation failure after the gate was requested.

## 8. Build Artifact Sequence

`build-artifact` now runs:

```text
Input State
→ Phase 3 validation
→ DOCX render
→ Structural DOCX validation
→ Optional Word Gate
→ Final structural validation when Word succeeds
→ Manifest
```

Without `--require-word`:

- Word availability is not checked.
- COM is not imported.
- Manifest records `{"status": "NOT_RUN", "availability": "not_checked"}`.
- Artifact status is `STRUCTURALLY_VALID` when DOCX validation passes.

With `--require-word`:

- Word gate runs on the DOCX generated in the same build.
- Success gives `WORD_VALIDATED`.
- Timeout/unavailable/failure gives artifact status `BLOCKED` and exit code `3`.

## 12. Security Controls

Implemented controls:

- DOCX output paths must stay inside the current workspace.
- Word input and output must be `.docx`.
- Word output cannot overwrite input.
- Word output is written through a temp file and atomically replaced only after success.
- Partial temp output is removed on failure or timeout.
- DOCX validator rejects unsafe package paths, external relationships, embedded executable-like parts, malformed XML, missing required parts, missing fields, and broken footnote linkage.
- Word automation uses subprocess timeout isolation.
- No internet access is required for tests.
- No macros are run.
- No global Word process kill is used.

Static scan:

```powershell
rg -n "\beval\(|\bexec\(|pickle|os\.system|shell=True|taskkill|WINWORD\.EXE|subprocess\.Popen|DispatchEx" src tests
```

Findings were expected, reviewed uses of `DispatchEx` in Word worker/availability, `subprocess.Popen` in runner, and test fakes. No `eval`, `exec`, `pickle`, `os.system`, `shell=True`, or `taskkill` usage was found.
